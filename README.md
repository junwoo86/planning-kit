# planning-kit — 전략기획 하네스 사용 매뉴얼

> 기획팀용 사용 설명서입니다. 이 킷이 *무엇이고*, *어떻게 시작하고*, *결과를 어떻게 읽는지*를 다룹니다.
> (AI 행동 거버넌스·작업 지도는 [CLAUDE.md](CLAUDE.md), 방법론 본문은 [docs/planning-harness-guideline.md](docs/planning-harness-guideline.md)에 있습니다.)

---

## 1. 이게 뭔가요

**planning-kit**은 개발 하네스(devkit)와 *분리된*, **필요할 때만 호출하는 상류(上流) 전략기획 도구**입니다.
가변 입력(녹취·문제+방향·"IPO 하자" 같은 큰 목표·러프 기획서)을 **적응형 인터뷰**로 받아,
흔들리지 않는 성공기준과 문제 정합성을 보장하는 **고품질 PRD(기획서)**를 만들어 개발팀에 넘깁니다.

이 킷의 목표는 "빨리"가 아니라 **"조용히 틀린 기획을 0으로"** 입니다 —
착수 후 정책이 비어 방향을 잃거나, 두 문제를 풀며 한 문제를 몰래 악화시키는 일을 막습니다.
**틀린 채 통과하는 것이 불완전한 것보다 나쁩니다.** 그래서 이 킷은 미흡하면 솔직하게 '미완/조건부'라고 말합니다.

---

## 2. 빠른 시작 (3단계)

1. **맥락 파일을 둡니다** → `inputs/<주제-slug>/`에 가진 자료를 넣습니다 (아래 §3).
2. **시작합니다** → 채팅에 **`/strategy-plan`** 입력 (또는 "전략기획 시작"·"기획 하네스"라고 말해도 됩니다).
   - 이미 진행 중인 기획이 있으면 알아서 **이어서** 진행합니다(처음부터 다시 묻지 않습니다).
3. **인터뷰에 답합니다** → 킷이 질문만 던지지 않고 *방안 1~3개를 장단점·추천과 함께* 제시합니다. 고르거나 고치면 됩니다.
   - 끝나면 `docs/planning/<slug>/`에 PRD와 단계별 산출물이 쌓입니다.

> ⚠️ **중요한 순간엔 반드시 멈추고 묻습니다.** 되돌릴 수 없는 일(외부 발행), 경계(인증·결제·개인정보),
> 토대(핵심 문제정의·성공기준·기획 의도 범위)는 *확신과 무관하게* 항상 확인을 받습니다.

## 3. 입력 준비 — `inputs/<slug>/`

가진 자료를 `inputs/<주제-slug>/` 폴더에 넣어두면 킷이 전부 읽고 시작합니다.
(slug = 기획 주제를 영문 kebab-case로. 예: 견적 자동화 → `inputs/quote-automation/`)

- 자료가 한 덩어리면 `inputs/quote-automation/context.md` 한 파일이면 충분합니다.
- 녹취·기존 기획서·메모 등 여러 개여도 됩니다 — 다 넣으세요.
- **자료가 없어도 시작 가능**합니다("IPO 준비" 같은 큰 목표만 있어도) — 킷이 인터뷰로 끌어냅니다.

**입력 4가지 유형** (어느 거든 같은 결과로 수렴합니다):

| 유형 | 가진 것 | 킷의 접근 |
|---|---|---|
| (a) 녹취/관찰 | 회의록·인터뷰 텍스트 | 테마를 묶어 문제·워크플로를 *귀납* |
| (b) 문제+방향 | "X를 Y로 풀자" | 전제·범위를 검증한 뒤 *확장* |
| (c) 제로베이스 | "IPO 하자" 같은 큰 목표만 | 목표에서 역산하며 *발산 인터뷰* |
| (d) 기존 문서 | 러프 기획서·정책 | 빈칸을 찾아 *갭 채움* |

> `inputs/` = **원본**(사람이 넣는 곳) · `docs/planning/` = **산출물**(킷이 만드는 곳). 둘을 섞지 않습니다.

## 4. 진행 파이프라인 (S0~S12)

킷은 아래 순서로 진행하며, **각 단계가 끝날 때마다 `docs/planning/<slug>/`에 결과 파일을 남깁니다.**
그래서 중간에 멈췄다 돌아와도 산출물만 보고 이어갈 수 있습니다.

| 단계 | 무엇을 | 산출 파일 | 지금 |
|---|---|---|---|
| S0–S1 | 입력유형·규모(tier) 확정 + 적응형 인테이크 | `00-intake.md` | ✅ 작동 |
| S2–S3 | 문제를 근본원인까지 **동결** + 성공기준 3층 | `01-problem-set.md`·`02-success.md` | ✅ 작동 |
| S4·S5 | 레퍼런스 인지 주입 / 워크플로·엣지케이스 | `03-…`·`04-…` | 🟡 초안(배선됨) |
| S6 | 해법 설계 + **정합성 매트릭스**(샛길 감지) | `05-coherence-matrix.md` | ✅ 작동 |
| S7·S8 | **참조 정합·교차 도메인** / 가정 | `06-…`·`07-…` + 참조 맵 갱신 | 🟡 초안(배선됨) |
| **S8 완료 → 보고서** | 진행분을 통합한 **검토 보고서** | **`REPORT.md`** | ✅ 작동 |
| **물음 G1** | "PRD 보완작업을 진행할까요?" | No → 보고서로 종료 / Yes ↓ | — |
| S10·S11 | 채택(잠정) + **PRD 산출**(이중 뷰) | `10-prd.html`·`10-prd-handoff.md` | ✅ 작동 |
| **물음 G2** | "HTML 목업(S9)을 진행할까요?" | No → '조건부' 종료(**Figma**) / Yes ↓ | — |
| S9·S12 | 목업 보강 → 채택 검증 → 개발 핸드오프 | `08-mockups/`·`11-handoff.md` | 🟡 초안(배선됨) |

**이 킷은 S12까지 직진하지 않습니다.** S0–S8을 진행해 **검토 보고서(`REPORT.md`)** 를 먼저 내고,
그 뒤 **PRD를 만들지(G1)**, **HTML 목업까지 할지(G2)** 를 *물어보고* 연장합니다.
기본 경로는 **목업은 기획팀이 Figma로 별도 진행** — 그래서 보통 보고서(+필요 시 PRD)까지가 킷의 몫입니다.
🟡 초안 단계는 파이프라인에 **호출 경로가 배선돼 있고 절차 골격이 있으나** MVP 실사용 검증 전이라 첫 실사용에서 고도화합니다. 산출이 비거나 얕으면 게이트가 정직하게 **'조건부'로** 표시합니다(§6).

> **파일 번호 ↔ 단계 매핑** (헷갈림 주의): 파일 번호는 *산출 슬롯 순번(0부터)*이라 대체로 **파일번호 = 단계번호 − 1** 입니다 — `00`=S1 인테이크 … `05`=S6 정합성 … `07`=S8 가정 … `11`=S12 핸드오프. 특히 헷갈리는 지점: **`03`=S4 · `06`=S7 · `07`=S8**. 목업 `08-mockups/`(S9)는 번호상 PRD(`10`)보다 앞이지만 *실행*은 G2 "Yes" 때만(나중 실행). 전체 표 = [`_state` 템플릿](.claude/templates/_state.md)의 단계 상태 표. (파일명은 도구가 하드코딩 참조하므로 변경 금지.)

## 5. 산출물 읽는 법 — `docs/planning/<slug>/`

- **`_state.md` (재개 앵커)** — *여기부터 보세요.* 단계별 진행 상태(완료/진행/대기), 현재 위치, 다음 할 일,
  그리고 **미해결 질문·결정 원장**이 한곳에 모여 있습니다. 어디까지 됐는지 이 한 파일로 파악됩니다.
- **`REPORT.md` (검토 보고서)** — *1차 산출물.* S0–S8 진행분을 통합한 의사결정 보고서: 요약·**신뢰성 스코어카드**·동결 문제·성공기준·정합성 결론·**교차 도메인 시사점**·검증 필요 가정·다음 결정(PRD 진행 여부). 목업/PRD 없이 여기서 멈출 수 있습니다.
- **`10-prd.html` (의사결정용 PRD)** — *G1에서 PRD를 진행한 경우.* 브라우저로 엽니다. 맨 위에 **판정(GO / 조건부 GO / NO-GO)**과
  **의사결정 신뢰성 스코어카드**(항목별 점수 · 신뢰성 종합 상/중/하 · 한계·주의 한 줄), 이어서 문제·설계 결정(채택/기각)·정합성·리스크·다음 단계.
  → *한눈에 "얼마나 믿어도 되는지 / 무엇을 조심할지"* 가 보이게 구조화돼 있습니다.
- **`10-prd-handoff.md` (개발 핸드오프용 PRD)** — 개발팀에 그대로 넘기는 명세(개발 하네스 입력 호환).
- 두 PRD는 **같은 내용의 두 표현**입니다(하나가 원본, 다른 하나가 뷰 — 따로 노는 게 아닙니다).

**도메인을 가로질러 보는 곳 = `docs/reference-context-map/`** (세션 시작 시 자동 생성):
- `finalized/` = **확정**(고정 제약 — 현재 보고서가 맞춤) · `open/` = **검토·컨셉**(참고 가능한 의견 — 보고서에 참고 반영) · `cross-domain-analysis.md` = 의존성·충돌·**선제 완성 우선순위**.
- 채우는 법 2가지: 킷으로 기획할 대상은 `inputs/<주제>/`에 올리면 자동 등재 / *참조만* 할 자료는 여기 `finalized·open`에 직접(헤더 한 줄). 자세히는 [inputs/README.md](inputs/README.md).

**Notion 링크 참조(`*.refs.md`)를 쓰려면 — Notion MCP 인증이 필요합니다:**
1. 채팅에 **`/mcp`** 입력 → Notion 서버 선택 → 브라우저 인증(OAuth)을 마칩니다.
2. 인증되면 킷이 `*.refs.md`의 Notion URL을 `mcp__claude_ai_Notion__fetch`로 읽어 참조에 반영합니다.
3. **미인증·권한없음이면 조용히 넘어가지 않습니다** — 해당 참조는 **`[확인불가]`** 로 표시하고(추측 금지), 진행 보고에 남깁니다.
> `finalized/`가 가리키는 Notion 문서는 **읽기전용**(킷이 수정하지 않음). 자세한 존 규칙은 [CLAUDE.planning-lite.md](CLAUDE.planning-lite.md) 참조.

## 6. 품질 게이트 — 왜 가끔 '조건부 GO'가 뜨나

이 킷은 **기계적으로 검사할 수 있는 것은 도구로 자동 판정**합니다(사람의 눈을 대체하는 게 아니라, *놓치는 걸* 막습니다).

- **정합성 검사** (`coherence-check`) — 어떤 기획 요소가 *동결된 문제 중 하나를 악화시키는데 대책이 없으면* 통과시키지 않습니다.
- **게이트 두 모드** (`completion-gate`):
  - **보고서 게이트**(`--mode report`) — S8 보고서 단계. 목업 의존 조건(⑤현업 채택 검증)은 **N/A(Figma 이관)** 로 빠집니다.
  - **종료 게이트**(`--mode full`) — 목업까지 진행해 PRD를 완성하는 경우. 아래 **8조건** 전체:
  ①문제 동결 ②성공기준 측정가능 ③미해결 악화 0 ④고위험 미해결질문 0 ⑤현업 채택 검증 ⑥모호표현 0("실시간·빠르게·직관적" 등) ⑦미기록 결정 0 ⑧검증필요 가정 누락 0.

판정은 세 가지입니다:

| 결과 | 뜻 |
|---|---|
| **완성 (COMPLETE)** | 8조건 모두 통과. '완성'을 선언해도 됩니다. |
| **조건부 (CONDITIONAL)** | 구조는 통과했지만, *측정가능성의 적절성*·*가정의 충분성*처럼 **사람 판단이 필요한 항목이 남음** → 독립 검토 후 완성. |
| **미완 (NOT COMPLETE)** | 빠진 정책·미해결 악화·열린 고위험 질문 등이 있음 → '조건부'로만 내보냅니다. |

> **'조건부'는 버그가 아니라 정상입니다.** 킷이 *모르는 것을 아는 척하지 않는다*는 신호입니다.
> 목업을 Figma로 별도 진행하는 기본 경로에서는 PRD가 '조건부'로 끝나는 게 정상이며(채택 검증은 Figma 이후), 준비 중 단계(S4·S5·S7~S9)의 산출이 없을 때도 게이트가 정직하게 '조건부'로 표시합니다.

## 7. 출처 태그 — 어디까지가 사실이고 어디부터가 AI 추론인가

PRD의 모든 줄에는 출처 태그가 붙습니다. **AI가 메운 부분을 항상 눈에 보이게** 하기 위함입니다:

- **출처**: `[입력]` 자료/사용자 직접 · `[AI보강]` AI 제안 · `[근본원인추론]` 5whys로 도출 ·
  `[현업검증]` 관찰/인터뷰로 대조 · `[참조맵:as-of 날짜]` 참조 컨텍스트 맥락 · `[물어봄]` 아직 미해결(추측 안 함).
- **확신도**: `[확실]` 독립 검증 2개+ · `[추정]` 신호 약함 · `[확인불가]` 근거 없음(이 위에 권고를 세우지 않음).

→ `[물어봄]`·`[확인불가]`가 많으면 그만큼 더 확인이 필요하다는 뜻입니다. `[AI보강]`·`[추정]`은 검토 1순위입니다.

## 8. 중단하고 나중에 이어가기

그냥 닫았다가, 나중에 다시 **`/strategy-plan`** 만 치면 됩니다.
킷이 `_state.md`를 읽고 *멈췄던 지점의 다음 할 일*부터 이어갑니다. 진행 상황은 모두 파일에 남아 있습니다.

## 9. 현재 범위와 로드맵

- **MVP 실증(코어)**: 문제 동결 · 성공기준 · 정합성 매트릭스 · **S8 검토 보고서** · PRD 산출 + 두 결정적 게이트(보고서/종료). 실사용 검증됨.
- **배선됨·초안(P2)**: 레퍼런스 탐색 · **참조 컨텍스트 맵 + 교차 도메인 분석**(성숙도 3그룹 확정/검토/컨셉) · 가정 관리 · (opt-in) 목업 UI 루프 — **파이프라인에 배선됐고 절차 골격이 있으나** 첫 실사용에서 고도화. 독립 검토 에이전트·활성화/종료 훅은 일부 DRAFT(미연결).
- **개선 방식**: 실제 기획(도메인)을 돌려보며 빈틈을 찾고, 도구를 고도화하거나 미구현 단계를 채웁니다. (작은 건부터.)

## 10. 지켜야 할 것 (거버넌스)

- **킷 자체(스킬·도구·규칙)는 임의로 바꾸지 않습니다.** 변경은 "무엇을·왜" 제안으로 만들고 사람이 승인합니다.
- 산출 문서는 `docs/` 아래에, 한 주제는 한 문서로(복사하지 말고 링크).
- 외부로 나가는 것(메일·게시·배포)은 한 번 나가면 회수 불가 — 실행 전 반드시 확인합니다.
