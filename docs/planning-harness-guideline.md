# 전략기획 하네스 지침 — 처음부터 흔들리지 않는 기획서(PRD)를 만든다

목적: 가변 입력(녹취·문제+방향·제로베이스 등)을 **적응형 인터뷰**로 받아, *흔들리지 않는 성공기준*과
*문제 정합성*을 보장하고 **근본원인**을 푸는 고품질 PRD를 생성해 개발 하네스(devkit)에 **아티팩트로만** 넘긴다.

헤드라인은 "빨리"가 아니라 **"조용히 틀린 기획 없이"** — 착수 후 정책이 비어 방향을 잃거나, 두 문제를 풀며
한 문제를 몰래 악화시키는 일이 **0**이 되는 것이 목표. 틀린 채 통과하는 것이 불완전한 것보다 나쁘다.

> **이 지침이 채우는 빈자리.** devkit은 "60% 초안 → 개발가능 명세(feature-spec)"에서 *시작*한다.
> 그 60%를 *만들기 전* 단계 — 올바른 문제 정의·성공기준·문제 정합성·비전↔실무 다리 — 가 통째로 비어 있다.
> 이 하네스가 그 상류를 담당하고, 산출 PRD를 devkit의 입력으로 넘긴다.

---

## 1. 성공 방정식 — 4대 불변 원칙

기획이 샛길로 새지 않게 잡아주는 고정된 기준. 모든 단계가 이 4원칙에 종속된다.

1. **흔들리지 않는 성공기준 (계층형 3층).** 성공 = ①**현업 채택**(실무자가 실제로 쓰는가) ← ②**핵심지표**
   (North Star + *출시 전 사전약속한 임계값*, 목표선 이동 금지) ← ③**정의한 문제 N개의 *근본원인* 해결 체크**.
   세 층이 **모두** 충족돼야 성공. 완료(기능 출고) ≠ 성공(문제 해결 + 채택).
2. **문제 정합성 (샛길 방지).** 동결한 문제집합을 기준점으로, 모든 기획 요소가 *모든 문제를 옳은 방향으로* 미는지
   매트릭스로 상시 검사. 두 문제를 풀며 한 문제를 악화시키면 멈춘다(§9).
3. **근본원인 해결 (증상 아님).** 5 whys로 문제를 근본원인까지 재정의하고, **해법이 증상이 아니라 근본원인을
   실제로 해소하는지**까지 판정한다. 증상만 가리는 해법(band-aid)은 미달로 표시한다.
4. **비전↔실무 양방향 다리.** 비전 입력엔 *현업 검증*을 강제 주입하고(대표가 못 보는 나무), 실무 입력엔
   *전략 승격*을 강제 질문한다(실무자가 못 보는 숲). 양쪽의 결손 축을 하네스가 메운다.

## 2. 작동 파이프라인 (S0~S12)

| 단계 | 이름 | 핵심 산출 |
|---|---|---|
| **S0** | 세션 시작 프로토콜 | 입력유형·참조맵 상태·적용강도(tier) 확정 |
| **S1** | 적응형 인테이크 | 유형별 진입 → 공통 산출로 수렴 |
| **S2** | 문제 정의 & 동결 | 근본원인까지 재정의된 **동결 문제집합** |
| **S3** | 성공기준 정의 | 계층형 3층 + 측정가능 지표 |
| **S4** | **레퍼런스 인지 주입** ★ | 실사용 검증 레퍼런스 + unknown-unknowns |
| **S5** | 워크플로/엣지 포착 | as-is/to-be 맵 + 엣지케이스 목록 |
| **S6** | 해법 설계 + 정합성 매트릭스 | 요소별 문제 영향(+/0/−) + 우선순위 |
| **S7** | 참조 정합성·교차 도메인 | 의존성·충돌·도메인별 권고·선제 완성 우선순위(해당 시) |
| **S8** | 가정·가설 관리 + 선례 | 검증 필요 가정 + 보강 조사 |
| **S9** | **목업 UI/UX 도출 루프** ★ | mock 화면 + 취합된 UI/UX 요구 |
| **S10** | 채택(adoptability) 선검증 | 현업 준비도(readiness) |
| **S11** | PRD 산출 + 무결성 게이트 | 이중 뷰 PRD(완성 판정) |
| **S12** | 개발 핸드오프 | devkit feature-spec 매핑 |

★ = 대표 추가 기둥. 단계는 순서가 기본이되, S4·S7·S8·S9는 *필요·비례*에 따라 깊이를 조절한다(§16 tier).

> **S8 보고서 체크포인트 + ASK 게이트(opt-in 연속).** 파이프라인은 S12까지 직진하지 않고 *선택적으로 연장*한다 —
> 어떤 단계도 제거하지 않되, 무겁고 외부도구(Figma)와 겹치는 하류를 ASK로 전환한다.
> ① **S8 완료 → 보고서(`REPORT.md`) 생성** = 1차 산출물(템플릿 `report.md`). 동시에 참조 컨텍스트 맵(§10) 갱신.
> ② **ASK G1 — "PRD 보완작업을 진행할까요?"** *No* → 보고서로 종료(참조 맵 확장 모드). *Yes* → S10·S11(PRD) 작성.
> ③ **ASK G2(PRD 마무리 시점) — "HTML 목업(S9)을 진행할까요?"** 기본은 *No*(기획팀이 Figma 와이어프레임으로 별도) →
> PRD를 '조건부'(목업/현업 readiness는 Figma 이관)로 종료. *Yes* → S9 목업 루프로 PRD 보강·readiness 검증 후 **S12까지 풀 진행**.
> 게이트 도구도 두 모드: `completion-gate --mode report`(S8 보고서 게이트, 목업/readiness 조건 N/A) vs `--mode full`(종료 8조건).

> **재개 가능성.** 각 단계는 `docs/planning/<slug>/`의 약속된 파일에 상태를 남긴다(`_state.md` = 진행 앵커: 단계별 done/in-progress/pending
> + 다음 액션). 작업이 중단됐다 돌아와도 *산출물만 보고* 이어간다 — 세션 시작 시 `_state.md`부터 읽는다. (상세 = 청사진 §6.3.)

## 3. 세션 시작 프로토콜 (S0)

본 인터뷰 *전에* 세 가지를 확정한다. 추측하지 않고 **묻는다**.

1. **입력 유형 분류** — 지금 가진 것이 무엇인가:
   - (a) **녹취/관찰 자료**(화자+텍스트) → *귀납 추출* 모드
   - (b) **문제 + 해결 방향** 제시 → *검증·확장* 모드
   - (c) **제로베이스**(예: "IPO" 같은 큰 목표만) → *발산 인터뷰* 모드
   - (d) **기존 문서**(러프 기획·정책) → *갭 채움* 모드
   - **혼합·모호 입력**: 유형이 섞이면(예: 녹취 + 기존 문서) 해당 유형별 진입을 *모두 돌려 병합*한다. 유형이 불분명하면
     동결 전에 *판별 질문*으로 먼저 좁힌다(추측 진입 금지).
   - → 모드가 달라도 **같은 산출(동결 문제집합 + 계층형 성공기준)으로 수렴**한다.
2. **참조 컨텍스트 맵 상태** — 최신 / 낡음 / **비어있음**. 비어있거나 낡으면 §10의 단언을 *하지 않고*
   "맵 미보유"로 플래그한다(없는 맥락을 그럴듯하게 메우지 않는다).
3. **적용 강도(tier·비례성)** — 이 기획의 전사 파급·비가역성·규모(§16). 작고 국소면 가벼운 흐름, 전사급이면 풀 프로세스.

## 4. 인터뷰 엔진 — 적응형·추천우선·의존순차·종료게이트

이 하네스의 인터뷰는 *질문 기계*가 아니라 *공동 설계자*다.

- **적응형 인테이크(S1).** 유형별로 다르게 시작한다: 녹취=테마 클러스터링으로 문제·워크플로 귀납 / 방향제시=
  전제·범위 검증 후 확장 / 제로베이스=목표에서 역산(Working Backwards)하며 발산 / 기존문서=빈칸을 플래그로.
- **추천 우선(질문만 던지지 않는다).** 이미 가진 맥락으로 *가장 합리적인 방안 1~3개*를 **장점·단점·리스크**와 함께
  제시하고 추천을 표시한다. 나쁜 선택지도 *왜 기각인지*와 함께 보인다. 사용자가 맨땅에서 고민하지 않게.
- **의존성 기반 순차.** 한 질문의 답이 다음 질문의 더 좋은 선택지를 바꾼다. 독립 질문은 묶고, 강한 의존이 있으면
  배치를 쪼개 *답을 받아 다음 질문을 재구성*한다(승인 피로 방지 + 정확도).
- **종료 게이트(§14)** 충족 전엔 인터뷰를 끝내지 않는다 — *가능성을 임의로 열어둔 채 진행하지 않는다.*
- **ASK 바닥(불변).** 비가역·경계(인증·권한·결제·PII)·토대(핵심 문제정의·성공기준·기획 의도 범위)는
  tier·확신과 무관하게 **무조건 멈추고 묻는다**. 판단 기준은 확신이 아니라 위험도(impact)다.
- **비대화형(원샷) 실행.** 사용자와 실시간 문답이 불가능하면 막지 말고 *진행하되*, 미해결을 `물어봄` 태그로 남기고 PRD를
  **'완성'이 아니라 '조건부'로** 낸다. 종료 게이트의 '완성'은 대화형에서만 허용한다(추측으로 메우지 않는다).

## 5. 문제 정의 & 동결 (S2)

기획의 기준점. 여기가 흔들리면 전부 흔들린다.

1. **증상·요청을 그대로 받지 않는다.** "X를 자동화해줘"는 증상일 수 있다. **5 whys**로 근본원인까지 내려간다.
   (devkit core `problem-solving`: 근본원인 → 프로세스/구조 수정 → 결과물 수정. 동일 철학.)
2. **현업 검증.** 추출한 문제를 현업 관찰/녹취(있으면)나 인터뷰로 *실제 그 문제가 맞는지* 대조한다.
3. **동결.** 문제집합을 명시적으로 확정한다. **이후 추가·변경은 '변경 요청'으로만** 한다(조용한 범위 이동 금지).
4. **문제마다 기록**: 문제 진술(근본원인 수준) · 영향받는 주체 · 현재 상태(왜 부족한가) · *안 풀면 드는 비용*.
5. **양방향 다리 — 전략 승격(여기서 주입).** 실무발 입력(예: "이 반복작업을 자동화")이면 동결 전에 *전략 승격*을 강제 질문한다:
   "이 작업이 어떤 더 큰 문제 / 프로세스 파괴로 이어지나?" — 근본원인이 한 층 위의 문제일 수 있다. (반대 방향 *현업 검증*은 S5·S10에서.)

**변경 요청 프로토콜 (동결 이후 문제 추가·변경).** 동결은 침범 금지지만 *틀린 동결*은 더 위험하다. 동결 후 문제집합을 바꿔야 하면:
① ASK 게이트(사람 승인) → ② 문제집합 재동결 → ③ **하류 강제 재실행**(S3 성공기준 · S6 정합성 매트릭스 재검증 · 영향받은 S9 목업) →
④ 무엇이 왜 바뀌었는지 provenance 기록. 조용히 끼워넣지 않는다.

> 동결된 문제집합 = §9 정합성 매트릭스의 **행(行)**. 성공기준 ③층의 평가 대상.

## 6. 성공기준 정의 (S3)

"무엇이 성공인가"를 측정 가능하게, 출시 전에 못박는다.

- **계층형 3층** 각각에 지표를 단다:
  - ③ **문제해결**: 각 문제의 *근본원인 해소*를 재는 **원인 지표**(증상 지표 아님). 구분법 — "이 지표가 좋아져도 근본원인이
    그대로면 문제가 재발하는가?" 재발하면 증상 지표이므로 불충분하다.
  - ② **핵심지표(North Star + 입력지표 3~5개)**: 핵심 가치를 한 지표로. **사전약속 임계값**("지표 X가 기한 Y까지
    Z만큼 안 움직이면 실패")을 *출시 전에* 선언한다 — 사후 목표선 이동(goalpost moving) 금지.
  - ① **현업 채택**: "실무자가 실제로 쓰는가"의 측정(활성 사용·재방문·대체 폐기율 등).
  - **베이스라인이 없으면**: 임계값을 지어내지 않는다 — *베이스라인 측정*을 선행 과제로 두거나 대체(proxy) 지표를 `추정` 태그로 쓴다.
- **모호 표현 규율.** "실시간·100%·빠르게·직관적" 같은 *측정 불가* 표현은 자동 플래그한다. "왜 모호한가 + 측정가능
  재정의안(SMART: 구체·측정가능·달성가능·관련·기한)"을 제시하고, **확정 전엔 미완**으로 둔다.
- **선행/후행 분리.** 선행지표(결과 전에 조기 신호를 주는 지표) + 후행지표(진짜 성과)를 함께 둔다.
- **non-goal 명시.** *무엇을 풀지 않는가*를 적어 범위 침범(scope creep)을 막는다.

## 7. 레퍼런스 인지 주입 (S4) — 당사자도 모르던 것을 인지시킨다 ★

문제 윤곽이 잡힌 뒤(S2 이후), *이미 실사용 검증된* 레퍼런스를 찾아 기획자의 인지를 넓힌다.

- **타이밍 게이트.** 문제가 정의되기 *전에* 레퍼런스를 찾으면 노이즈다. S2 동결 이후에 발화한다.
  (NN/g upfront-vs-continuous discovery; Reforge: 경쟁조사는 *연구 단계의 구성요소*이지 사전 단계가 아님.)
- **무엇을 찾나.**
  - 같은/유사 문제를 푼 제품·시스템(업계 안팎 모두 — Design Sprint *Lightning Demos*: "내 업계 밖에서도 배울 점").
  - **교차 도메인 유추**: 다른 산업이 유사 문제를 푼 방식(원거리 유추가 새로움을 만든다 — arXiv 1706.05585).
  - 경쟁/선례 **teardown**: 핵심 기능·설계 결정·*왜 그렇게 했는지*.
- **'실사용 검증' 필터 + 증거 기준.** vaporware·마케팅 발표가 아니라 *실제 채택 증거*가 있는 것만 신뢰한다. S4는 웹/리서치
  도구로 탐색한다(깊으면 devkit `research`/`deep-research` 위임). 증거 기준: **독립 채택 신호 2개 이상 = `사실` / 1개 = `추정`
  / 0개 = `확인 불가`**. 확인 불가 레퍼런스 위에 권고를 세우지 않는다.
- **레퍼런스 부재 처리.** 신규·선례 없는 문제면 *아무것도 안 나올 수 있다*. 진행을 막지 않되 '선례 없음 = 불확실성↑'로
  플래그하고 §11 가정을 늘린다. *검증된 듯* 진행하지 않는다.
- **unknown-unknowns 드러내기.** "당사자도 *모른다는 걸 모르는* 부분"을 의도적으로 묻는다(Rumsfeld Matrix의
  unknown-unknown 칸; Sutcliffe & Sawyer, IEEE RE'13).
- **카고컬트 차단(필수 게이트).** "저기서 먹혔으니 우리도"는 **금지**. 모든 레퍼런스 통찰은 §9 정합성 매트릭스 +
  동결 문제집합 + 전사 맵(§10)을 **통과해야** 채택 후보가 된다 — *우리* 문제를 실제로 풀고 *우리* 맥락에 맞는가.
  기능 모방(feature parity)은 차별성 파괴·자원 낭비·맥락 무시를 부른다(Nulab, FeatureBase, nickmcevily, tamm).
  **레퍼런스는 인지를 넓히되 게이트를 우회하지 못한다.**

## 8. 워크플로 & 엣지케이스 포착 (S5) — '실제 쓰는 시스템'의 핵심

현업이 실제로 쓰려면 happy path만으로는 안 된다. *실제 실행*을 그린다.

> 양방향 다리 — **현업 검증 방향**: 비전발 입력이면 여기서 as-is·엣지·채택가능성으로 *현업 검증*을 강제한다(대표가 못 보는 나무).

- **as-is 매핑(실제 실행).** 이론상 절차가 아니라 *현재 진짜로 일어나는* 흐름을 단계로 그린다 —
  비공식 우회·스프레드시트 편법·반복 확인 같은 *비공식 관행*까지 포함한다(signavio/kissflow). 자료가 부족하면
  job shadowing/관찰 또는 인터뷰로 채운다(인터뷰가 못 잡는 뉘앙스를 관찰이 잡는다 — NN/g 서비스 블루프린트).
- **to-be 매핑.** 개선된 미래 흐름 + 전환(to-do) 액션.
- **엣지케이스 강제 질문.** 각 단계마다 명시적으로 묻는다: *뭐가 잘못되나 / 오프라인일 때 / 권한 만료 / 중단 후
  재개 / 부분 실패 / 동시 실행 / 빈 입력*. 에러·예외 상태를 빼먹는 것을 구조적으로 차단한다(figr).
- 산출: 각 화면/영역의 **상태**(초기·로딩·빈·성공·에러·권한없음)와 **예외 목록**. → feature-spec(fs§5·fs§8)로 매핑.

## 9. 해법 설계 + 문제 정합성 매트릭스 (S6) — 샛길 감지 엔진

대표의 시그니처. *기획 요소가 추가될 때마다* 모든 문제가 옳은 방향인지 검사한다.

**매트릭스(살아있는 원장).** 행=동결 문제, 열=기획 요소. 각 칸 = **개선(+) / 무영향(0) / 악화(−)**.
요소를 추가·변경할 때마다 갱신한다(devkit `open-questions` 원장 패턴 재사용).

| 문제 \ 요소 | 요소 A | 요소 B | … |
|---|---|---|---|
| 문제 1 | + | 0 | |
| 문제 2 | + | **−** (완화책 or 플래그) | |
| 문제 3 | 0 | + | |

**악화(−) 칸 규칙 — "대안 먼저, 안 되면 하드게이트".**
1. 악화가 감지되면 AI가 먼저 **악화 없는 대안 1~3개**를 제시한다(2차 효과로 숨은 악화도 발굴 — "그다음 뭐가?"를
   재귀적으로; TRIZ 모순 매트릭스로 "X 개선↔Y 악화"의 묘책 탐색).
2. 대안으로 해소되면 채택. 해소 안 되면 **완화책**(노브·탈출구·후속작업)을 칸에 명시한다.
3. 완화책도 없으면 PRD를 **조건부/블로킹**으로 표시하고 **사람 결정을 강제 + 기록**한다 — 조용히 넘어가지 않는다.
   이것이 "리스크를 미리 잡아 가이드하는 든든한 동료"의 구체 행동이다.

**상호배타 문제(요소가 아니라 *문제끼리* 충돌).** 두 동결 문제가 본질적으로 충돌해(문제 1 해결이 반드시 문제 2를 악화시키고,
*어떤 요소로도 둘 다 +가 불가능*) 대안이 원천적으로 없는 경우 — 이건 완화책 대상이 아니다. **사람에게 재범위화(re-scoping)를
강제**한다: 한 문제를 우선순위에서 내리거나 분리하고 그 결정을 기록한다. (정합성 매트릭스로도 못 푸는 유일한 경우 — 숨기지 않는다.)

**우선순위(다수일 때).** 문제·해법이 여러 개면 경량 **RICE/ICE**(impact·reach·confidence·effort)로 정량 정렬하고,
*가중치를 평가 전에 고정*한다(역설계 방지). 전사 파급(§10)과 결합해 "어떤 문제부터"를 정한다.

## 10. 참조 컨텍스트 맵 (S7) — 도메인을 가로질러 숲을 보는 엔진

대표의 "숲" 초능력을 *재사용 자산*으로 인코딩한다. **특정 목적("전사")으로 단정하지 않고**, 각 도메인 기획(`inputs/<도메인>` →
S0~S8)에서 *추출된 산출물의 점진 확장 모음집*으로 둔다. **이 지침은 자리·포맷·사용규칙만 규정**하고, 맵 내용은 도메인 기획이
하나씩 끝날 때마다 채워진다(빈 맵을 미리 그럴듯하게 메우지 않는다).

**위치(SSoT) = `docs/reference-context-map/`** (모음집 디렉토리):
- `index.md` — 도메인 목록·신선도·링크 · `<domain>.md` — 도메인별 요약(문제집합·성공기준·핵심 결정·의존 도메인·열린 이슈) ·
  `cross-domain-analysis.md` — **교차 도메인 분석 보고서**(아래).
- 템플릿 = `.claude/templates/reference-context-map.md` · `cross-domain-analysis.md`.

**입력 2종 구분.** (1) *기획별 새 인풋*(S1 가변 인테이크) vs (2) *상시 참조 맥락*(이 맵, ambient — 다른 도메인 산출의 누적).

**성숙도 3그룹 + 역할(실무 분류).** 도메인 PRD는 성숙도로 나뉜다 — **확정 / 검토 / 컨셉**. 위치는 역할로 갈린다:
- **확정(finalized)** → `reference-context-map/finalized/`. **역할 = 고정 제약(constraint)** — *현재 작성 중인 도메인 보고서가 반드시 맞춰야* 한다. 방향 수정 제안 대상이 아니며, 충돌 시 *열린 쪽*을 조정한다(확정 변경은 변경 요청 프로토콜로만).
- **검토·컨셉(open)** → `reference-context-map/open/`. **역할 = 참고 가능한 '의견'** — 구속력 없이 *현재 작성 중인 도메인 보고서에 참고용으로 반영*한다. 검토는 정합성·완전성 검토 + 교차 영향, 컨셉은 **적극적 방향 수정 제안**(양방향 가능; 단 확정 제약·동결 문제집합·정합성 게이트는 지켜야 채택 후보).
- **확정되면 `open/` → `finalized/`로 옮긴다**(그때부터 제약이 됨). 통합 PRD는 도메인별로 쪼개 status 판정(모호하면 ASK). 구조 점검 = `ref-map-lint`.

**맵 등재 2경로(목적으로 갈린다).** *"이 도메인을 킷으로 기획/보고서 생성할 건가?"* —
(1) **응(작업 대상)** → 사람은 원본을 `inputs/<주제>/`에만 올리고 킷이 처리해 등재(`source: 킷`). (2) **아니, 참조만**(이미 확정됐거나 다른 데서 진행 중) → 사람이 맵 경로(`finalized/`·`open/`)에 직접 등재(`source: 참조`, 헤더 한 줄). 킷은 `source: 참조` 항목을 임의로 덮어쓰지 않는다.

**교차 도메인 분석 보고서(검토보고서).** 맵의 도메인들을 가로질러 판정한다 — ① **의존성/영향도 매트릭스**(확정 열은 제약 표시) ·
② **충돌 지점**(확정 vs 열림이면 열린 쪽 조정) · ③ **도메인별 방향 수정 제안**(열린 도메인만 — 확정은 제외) ·
④ **선제 완성 우선순위**(어떤 *열린* 도메인을 먼저 확정하면 다른 도메인 불확실성·재작업이 줄나 — 피의존·leverage·비가역).
새 도메인이 S8 보고서까지 끝나거나 status가 바뀔 때마다 갱신한다.

**신뢰성 통제(필수) — 낡은 맵은 '자신 있게 틀린' 가이드가 된다.**
- 각 항목에 **[출처·수집일·confidence] 태그** + **2개 이상 출처 교차검증**.
- **신선도 게이트**: 기본 **6개월**(변화 잦은 영역은 3개월) 지난 항목은 "재확인 필요" 플래그. 맵은 **'as-of 증거'로만** 다루고 *단언하지 않는다.*
- **민감정보 경계**: 인원·전략·성과 등은 노출 대상을 통제한다.

**사용 2모드.**
- **방어(정합성 체크)** — *비례 적용*: 고파급 기획만 풀체크. 이 도메인이 다른 도메인 프로세스·데이터·정책을 악화시키는가, 숨은 리스크가 있는가.
- **발산(거시 묘안 제안)** — *상시 on, 라벨된 '선택적 인사이트'*: 거시 관점 묘안·새 관점·재구성을 저마찰로 제시(기획자 취사). scope 강박은 라벨링으로 방지.

**산출은 조건부(해당될 때만).** ① 숨은 리스크 회피 ② 시너지 안 ③ 파급효과(leverage) 큰 안 ④ 거시 묘안·새 관점. 매번 전부가 아니라 *해당되는 것만* 낸다.

## 11. 가정·가설 관리 + 선례 보강 (S8)

- **가정 매핑(Assumption Mapping, Bland).** 기획이 깔고 있는 가정을 *desirability(원하나)·feasibility(되나)·
  viability(해야 하나)* × *Known/Unknown × Important*로 분류한다. **Unknown + Important** 가정은 PRD의
  **'검증 필요 가정' 섹션**으로 빼고, 위험한 것은 실험·조사로 검증한다(가정한 채로 큰 베팅하지 않는다).
- **선례·경쟁 조사 연계.** S4에서 찾은 레퍼런스를 해법 설계에 반영하되, 깊은 조사가 필요하면 devkit
  `research`/`deep-research`로 위임(다출처·인용·사실/추정 분리).

## 12. 목업 기반 UI/UX 도출 루프 (S9) — Shift-Left ★

대표 고통("PRD만 보다 개발 후 수정 폭증 → 기간 증가")의 직접 해법. **변경을 기획 단계(싼 곳)로 끌어온다.**

**왜.** 변경비용은 늦게 잡을수록 기하급수로 비싸다(Boehm 20~100x; agile에선 곡선이 덜 가파르나 원리는 유효 —
Mountain Goat/Agile Modeling 균형). 텍스트 PRD로는 안 드러나는 *암묵 요구*가 "돌아가는 것 같은" 화면을 보면
튀어나온다(IKIWISI, "보면 안다" — Wiegers). 프로토타이핑은 요구 변동을 줄인다(Springer, IET SLR).

**루프.**
1. **목업 생성** — mock 데이터로 *돌아가는 것 같은* 화면을 임의 생성. 산출은 **자족적 단일 HTML 파일**
   (`docs/planning/<slug>/08-mockups/`)로 경로 제시. **`docs/design-system/`에 디자인시스템 문서가 있으면 그 톤앤매너를 참조**,
   없으면 중립 lo-fi(흑백·플레이스홀더). *적정 충실도*로 시작 — lo-fi 먼저; hi-fi는 픽셀·색상 bikeshedding과 한 디자인 고착을
   부른다(fidelity trap. Shape Up 'fat marker'). 필요 시 Wizard-of-Oz(가짜 자동)로 복잡 상호작용도 흉내.
2. **기획자 반응 수집** — 화면을 *눈으로 보고* "여길 이렇게 고치자"는 **수정 프롬프트**를 준다.
3. **자동 취합 + 충돌 검사** — 프롬프트들을 UI/UX 요구로 수집·중복제거하고, **프롬프트끼리 모순**("X를 맨 위로" ↔ "X 제거")이
   없는지 검사한다. 충돌은 취합 전에 기획자에게 ASK로 올린다.
4. **정합성 재검** — 취합된 변경(수정/추가/제거)을 §9 매트릭스 + 동결 문제집합 + 전사 맵에 다시 통과시킨다.
5. **PRD 반영** — 통과한 변경을 **PRD(SSoT)에 반영**한다. 필요하면 목업을 재생성해 다시 본다.

**자명한 UI 기준.** "30초에 설명 없이 이해"는 채택(§13)의 핵심. Nielsen 10 휴리스틱(특히 *recognition > recall*,
상태 가시성, 현실 언어 일치, 에러 예방)과 Krug "Don't Make Me Think"(자명·자기설명)를 목업 평가 잣대로 쓴다.

**리스크 규칙(반드시).**
- **R-목업1 외형 ≠ 구현가능성** — 목업엔 **"외형 전용·구현 검증 안 됨" 태그**. 목업이 암시하는 기능은 *조용히
  약속하지 않고* 추후 devkit tech-planning의 타당성 판정으로 라우팅한다.
- **R-목업2 충실도 함정** — 목업은 *버릴 수 있는 도출 도구*다. 최종 디자인이 아님을 명시한다.
- **R-목업3 UI발 스코프 팽창** — 목업 보다 생긴 새 욕심도 *문제집합·정합성 게이트* 통과 필수. 새 문제면 '변경 요청'.
- **R-목업4 PRD↔목업 drift** — 목업은 *뷰/프로브*이지 제2의 SSoT가 아니다. **PRD가 SSoT**이고, 프롬프트는 PRD를
  갱신한다(Specification by Example; SSoT 동기화).

## 13. 채택(adoptability) 선검증 (S10) — last-mile을 기획에서 막는다

POC가 현업에 안 닿는 6대 원인(데모데이터·통합현실·조직마찰 등)과 last-mile 마찰을 *기획 단계에서* 선검증한다.
PRD에 **'현업 준비도(readiness)' 섹션을 필수**로 둔다:

- 실무자가 **설명 없이 즉시** 이해하는가 — *측정*: 첫 사용자 n≥5명이 도움 없이 핵심 과제를 30초 내 완료, 성공률 ≥80%(미달 시 미완).
- 기존 워크플로(§8 as-is)에 *자연스럽게 끼는가*, 아니면 별도 행동을 강요하는가.
- 운영·교육 준비(누가 굴리고 누가 가르치나), 단계적 롤아웃 계획(빅뱅 금지).
- **채택 안 될 PRD는 '미완'으로 판정한다**(devkit 완료기준 패턴).

## 14. PRD 산출물 — 이중 뷰 · 무결성 게이트 · 출처 태그 (S11)

**이중 뷰(같은 SSoT, 두 표현).**
- **의사결정용 = HTML** — Executive Summary(GO/조건부GO/NO-GO) 우선, 정합성 매트릭스·트레이드오프 시각화.
  devkit `decision-doc` 구조(Executive Summary → 문제 → 설계 결정(채택/기각) → Design Detail → Risk → Next Step).
  - **의사결정 신뢰성 스코어카드(Executive Summary 직하).** 대표가 *한눈에 믿어도 되는지·무엇을 조심할지* 판단하게 하는 요약 표:
    평가 항목(근본원인 정확도·근거 충분성·다각도 검토·정합성(악화 처리)·채택 실현가능성)별 점수 + **신뢰성 종합(상/중/하)** +
    **한계·주의 한 줄**(미검증 가정·blocking·열린 고위험Q 요약). 종합 등급은 종료 게이트 판정(완성/조건부/미완)과 *일관*한다 —
    점수·코멘트는 정성 판단(사람/LLM), 도구(`completion-gate`)는 *등급만* 구조적으로 보조한다(정직성 원칙). 시각요소 = 판정 신호등 ·
    정합성 +/0/− 색상 히트맵 · 상단 리스크 핀.
- **개발 핸드오프용 = md** — devkit feature-spec 호환(§15 매핑).

**출처·확신도 태그(AI 추측을 항상 가시화).** 모든 줄에 두 축을 단다 —
**출처**: `입력` / `AI보강` / `근본원인추론` / `현업검증` / `참조맵(as-of 날짜)` / `물어봄(미해결)`,
**확신도**: `확실`(독립 검증 2+) / `추정`(신호 1·약함) / `확인불가`(근거 없음 — 이 위에 권고를 세우지 않는다). 확신도는 S4 §7의 사실/추정/확인불가
3등급을 PRD 전반으로 일반화한 것이며, 스코어카드 '근거 충분성'의 입력이 된다. 제로베이스일수록 추론 비중이 커지므로 태그가 더 중요하다(silent-wrong 방지).

**무결성 게이트(셀프 + 고위험 독립검토).** 출고 전 technical-review **4축**을 셀프 체크리스트로 통과한다:
빠진 정책·예외·전제 / 모호한 요구(측정불가) / 문서 간 정합성 / 구현 리스크. 고위험(T2) 기획은 2차 모델 또는
사람 **독립 검토**(devkit `cross-check`·`spec-reviewer`)를 추가한다.
> 판정 분담: *결정적 검사*(고정 금지어 목록·매트릭스 악화 칸·미기록 결정 등 값/구조가 명확한 것)는 도구로 자동 판정하고,
> 열린 의미적 모호성은 **독립 검토**로 잡는다 — 결정적 게이트가 의미 판단까지 책임지는 *척하지 않는다*.

**독립검토 트리거 — 단일 정의(T5, 이 절이 SSoT).** *언제* 독립검토(`coherence-auditor` veto 또는 2차 모델/사람)가 필요한가:
- **T2 = 다음 중 1+**: 경계(인증·권한·결제·PII) · 비가역(외부 발행·되돌릴 수 없는 합의) · **blocking 잔존**(미해결 악화) · **미검증 핵심 가정**. → **auditor veto 필수**(통과 = 4축 blocking 0 + veto 미발동).
- **T1 = 권장**(미수행 시 신뢰성 종합 **상한 '중'**). **T0 = 불요**.
- 독립검토를 마치면 `_state.md`에 **`[독립검토=완료]`** 표식을 남긴다 → `completion-gate`가 이를 읽어 **REVIEW를 PASS로 승격**한다(표식 없으면 REVIEW 유지). coherence-auditor·prd-compose·reference-coherence는 이 정의를 참조한다(중복 정의 금지).

**미해결 누적 임계 — 단일 정의(T4).** **토대 3축**(핵심 문제정의·성공기준·기획 의도 범위)에 걸린 미해결이 **1건+** 이면 해당 하류는 `_state` **`blocked`·'미완'**이다 — 도구는 cond3(blocking 악화)·cond4(고위험 open-Q)로 FAIL 처리한다. **비토대** 미해결은 가정 후 명시 진행. *보조 강등*: med open **≥5** 또는 **토대 핵심줄 `[확인불가]` 1+** 이면 신뢰성 종합 **'하'** 자동 강등(completion-gate가 01·02 스캔).

**종료(완료) 게이트 — "가능성을 임의로 안 열어둔다".** 아래가 *전부* 충족되기 전엔 **미완**:
1. 문제집합 동결 · 2. 계층형 성공기준 측정가능 확정 · 3. 정합성 매트릭스 **미해결 악화 0** ·
4. 고위험 open-question **0** · 5. 채택 readiness **통과**(존재가 아니라 §13 측정 합격) · 6. 모호표현 **0**(고정 금지어 기준) ·
7. 미기록 결정 **0** · 8. **Unknown+Important 가정이 '검증 필요 가정'에 모두 등재(누락 0)**.

## 15. 개발 핸드오프 계약 (S12) — devkit으로 아티팩트만 넘긴다

상류 PRD는 devkit `feature-planning`의 **입력**이 된다. 명시 매핑:

| 상류 PRD | → | devkit feature-spec |
|---|---|---|
| 문제집합·목적 | → | §3 기능 목적·non-goal |
| 워크플로(to-be) | → | §4 유저 플로우 |
| 화면 상태(§8) | → | §5 화면 상태 |
| 정책·정합성 매트릭스 결정 | → | §6 정책·규칙 |
| 엣지·예외(§8) | → | §8 예외 케이스 |
| 성공기준 | → | §10 완료 기준 |
| 미해결 가정·악화 | → | open-questions 원장 |

> 표기: 위 표의 **우측 §N = devkit feature-spec 섹션**, 본 지침 본문의 §N = *이 지침*의 섹션. 혼동 방지를 위해 외부 참조는 `fs§N`으로 쓴다.

빈틈은 추측으로 메우지 않고 **open-questions로 넘긴다.** devkit은 이 PRD가 어디서 왔는지 몰라도 된다 —
접점은 *아티팩트(PRD)뿐*이다(느슨 결합).

## 16. tier 운영 (impact 기반 · devkit 차용)

| Tier | 적용 | 동작 |
|---|---|---|
| **T0** | 작고 국소 | 가벼운 흐름(핵심 단계만), 전사 맵=가벼운 스캔 |
| **T1** | 중간 | 짧은 인터뷰 + 핵심 게이트 |
| **T2** | 전사 파급·비가역·IPO급 | 풀 인터뷰 + 전사 정합성 풀체크 + 독립 검토 + **사람 승인 게이트** |

**tier 판별(3문).** ① 비가역·경계(인증·결제·PII)에 닿나? ② 2개 이상 팀에 영향(전사 파급)? ③ 큰 베팅(예산·기간·전략 방향)?
— 하나라도 '예' → **T2**. 셋 다 아니고 한 화면/한 흐름 국소 → **T0**. 그 사이 → **T1**.

**core ASK 바닥은 tier 무관 불변** — 비가역·경계(인증·결제·PII)·토대(문제정의·성공기준·기획 의도)는 T0에서도 무조건 ASK.
tier는 *그 위*의 자율 폭만 정한다.

## 17. devkit 공존 & 거버넌스

- **분리·선택적.** 별도 킷(자체 지침/스킬/훅/에이전트/도구). devkit과 코드/게이트 결합 0. 평소 dev 작업엔 호출 안 하면
  작동 안 한다(오버헤드 0). 활성화 = 명시 호출 기본 + *강한 전략·전사·혁신 신호* 감지 시 조용한 제안(거부 가능).
- **drift 방지.** devkit common(ask-and-fallback·tier·출처태그·doc-quality 철학)을 **복사하지 않고 참조/정렬**한다.
  한 주제 = 한 SSoT.
- **하네스 변경은 제안+사람 승인.** 이 지침·킷을 AI가 임의 수정하지 않는다. 변경은 why+diff+근거 제안으로 만들고
  사람이 승인·머지한다(propose-harness-change/creator). 실제 스킬/훅 컴포넌트 설치는 사람 몫.
- **문서 위치.** 산출 문서는 `docs/` 아래. 한 주제 한 문서, 복사 말고 링크.

## 18. 워크드 예시 — 가상 사내 자동화 기획 1개 관통

> 가상 시나리오: 영업팀이 "견적서 작성을 자동화해달라"고 요청. (실제 데이터·수치 아님, 절차 시연용.)

- **S0** 입력유형=(b)문제+방향 · 참조맵=비어있음(플래그) · tier=T1(국소→중간).
- **S1** 검증·확장 모드 진입.
- **S2 문제 동결(5 whys).** "견적서 자동화" → 왜? 작성 30분 → 왜 오래? 제품·할인 규칙을 매번 수기 확인 →
  왜? 규칙이 흩어져 있음(SSoT 부재) → **근본원인 = '가격/할인 규칙의 단일 출처 부재'.**
  동결 문제집합: ①작성 시간 과다 ②규칙 불일치로 오견적 ③신입이 규칙을 못 찾음.
- **S3 성공기준.** ③근본원인 해소=규칙 SSoT 구축+참조 / ②핵심지표=오견적률(현재 측정→임계값 "출시 8주 내 1%↓")
  / ①채택="영업 80%가 도구로 견적 작성"(사전약속).
- **S4 레퍼런스.** 실사용 검증된 CPQ(Configure-Price-Quote) 패턴 teardown → "규칙 엔진 + 승인 워크플로"가 정석임을
  인지(당사자는 'CPQ' 개념 자체를 몰랐음 = unknown-unknown). **카고컬트 게이트**: CPQ의 복잡한 승인플로는 우리
  문제집합(①②③)에 과함 → 규칙 엔진만 채택, 승인플로는 기각(우리 맥락).
- **S5 워크플로/엣지.** as-is: 엑셀 단가표 열기→수기 계산→PDF. 엣지: *단가표가 어제 바뀐 경우 / 비표준 할인 요청 /
  오프라인*. → 상태·예외 목록화.
- **S6 정합성 매트릭스.** 요소 "규칙 엔진 자동적용"이 ①+ ②+ 이지만 ③신입에 대해 "엔진이 블랙박스라 학습 저해(−)".
  → **대안**: 엔진이 *적용 근거(어떤 규칙이 왜)*를 노출 → ③도 + 로 전환. (악화 해소.)
- **S7 전사 정합성.** tier=T1이라 방어 체크는 *가벼운 스캔*. 참조맵 비어있음 → 단언 보류, "재무팀 단가 SSoT와 충돌 가능"만 *플래그*.
  **발산(상시 on, 선택적 인사이트)**: "규칙 SSoT는 재무·법무 견적에도 재사용 가능 — 전사 단가 SSoT로 키울 여지"(당장 채택 아님, 기획자 취사).
- **S8 가정.** "영업이 새 도구로 갈아탄다"=Unknown+Important → '검증 필요 가정'. 파일럿으로 검증 예정.
- **S9 목업 루프.** 견적 화면 HTML mock 생성 → 기획자가 "할인 근거를 칩으로 보이게" 프롬프트 → 취합 → 정합성 재검
  (③ 강화) → PRD 반영. (외형 전용 태그 / 규칙엔진 성능은 tech-planning으로 라우팅.)
- **S10 채택.** readiness: 신입이 설명 없이 핵심 과제 30초 완료(n≥5, ≥80%)? 기존 엑셀 습관 대비 마찰? 단계 롤아웃(1개 팀 파일럿).
  *채택은 파일럿 전까지 잠정 — S8 미검증 가정과 연결.*
- **S11 PRD.** 이중 뷰 + 출처태그. 무결성 게이트: 악화 0·모호 0·미기록 0 통과. **그러나** 고위험 open-question(재무 SSoT 충돌)이
  열려 있고 영업 채택은 미검증 가정 → 종료 게이트 조건 4·5 **미충족** → **'완성' 아님, '조건부 GO'**(파일럿·재무 확인 후 완성).
  *게이트가 작동해 조용한 통과를 막은 예 — 이게 정상이다.*
- **S12 핸드오프.** feature-spec으로 매핑, "재무 단가 SSoT 충돌"·"영업 채택 가정"은 open-questions로.
